# X.509 Certificate Support

mmhashchainsigs supports three trust models. The wire format, hash
chain, and signature algorithm choices are independent of which model
is used.

## Signing algorithms

| Algorithm | Signature size | Notes |
|-----------|----------------|-------|
| Ed25519 (PureEdDSA) | 64 bytes | Default. RFC 8032. |
| ECDSA P-256 + SHA-256 | 70–72 bytes (DER) | RFC 6979 deterministic ECDSA recommended. Useful when policy or PKI tooling does not yet handle Ed25519 certs. |

Both algorithms are accepted as raw PEM keys, in X.509 certificates, and
in CA-bundle chain validation. The on-wire SD format does not encode
the algorithm — the verifier discovers it from the loaded key/cert.

## Wire format additions

When `embedcert=on` is configured on the signer, INIT and CONTINUE lines
carry one extra SD element after the chain-metadata element:

```
[mmhashchainsigs@32473 t="I" q="1" h="..." f="..."]\
[mmhashchainsigs-cert@32473 cert="<base64 DER>"]\
[mmhashchainsigs-hdr@32473 ...]MSG
```

The cert SD is **not** part of the hashed payload. The verifier strips
both `mmhashchainsigs@32473` and `mmhashchainsigs-cert@32473` before
reconstructing the chain. Ordinary MSG and SIG lines never carry the
cert SD — it appears only on chain-opening frames.

DER size budget: the SD-element formatter accepts up to 4096 bytes of
DER (`HCS_CERT_DER_MAX`), which covers any realistic CA-issued
Ed25519/ECDSA leaf. Typical certs are 300–800 bytes DER.

## Fingerprint

The 32-byte fingerprint carried as `f="<hex>"` on INIT and CONTINUE
lines is SHA-256 of the raw public-key bytes:

- Ed25519: SHA-256 of the 32-byte raw public key.
- ECDSA P-256: SHA-256 of the 65-byte uncompressed point
  (`0x04 || X || Y`).

When a verifier is in CA-bundle mode, the embedded cert is
chain-validated first; then its pubkey fingerprint is computed and
checked against the `f=` field of the INIT line.

## Verifier behavior

| Mode | Source of pubkey | Validity dates | Revocation |
|------|------------------|----------------|------------|
| `--publickey` | the PEM file | n/a | n/a |
| `--cert` (pinned leaf) | the cert's pubkey | not enforced — pinning supersedes them | n/a |
| `--ca-bundle` | the embedded cert | enforced by OpenSSL chain validation | optional `--crl-file` |

Revocation is optional in verifier-only and never queried over the
network. `--crl-file` loads a static CRL file and sets the
`X509_V_FLAG_CRL_CHECK | X509_V_FLAG_CRL_CHECK_ALL` flags so the entire
chain is checked. OCSP is intentionally not supported: the verifier is
designed to run on stored log data, possibly offline, and a network
fetch would make audit verification a hard runtime dependency on
external infrastructure.

The rsyslog module does **no** revocation checking — by design, the hot
path stays purely local.

## Cert rotation

mmhashchainsigs has no built-in cert hot-reload. To rotate:

1. Stop rsyslog gracefully (so the state file flushes a final SIG over
   the old chain).
2. Replace `mmhashchainsigs-cert.pem` (and the private key if rotating
   keys too).
3. Start rsyslog. The new chain begins with an INIT carrying the new
   fingerprint, and (if `embedcert=on`) the new cert.

A verifier in CA-bundle mode automatically picks up the new cert from
the new INIT line. Pinned-cert verifiers need their `--cert` argument
updated.

## Operational notes

- **Algorithm consistency.** Use the same algorithm for the private
  key and the cert — the loader rejects a mismatch.
- **Self-signed certs as a stepping stone.** `selfsign` mode of
  `mmhashchainsigs-keygen.sh` produces a self-signed leaf you can pin
  with `--cert` today and replace with a CA-issued cert later without
  changing wire format or fingerprint logic.
- **RSA is unsupported.** Loading an RSA private key, RSA pubkey, or
  RSA cert pubkey returns an error at startup. There is no fallback —
  audit signing demands deterministic, fast verification and small
  signatures; ECDSA P-256 covers any compliance requirement RSA would
  satisfy.
- **Header capture is unchanged.** The `[mmhashchainsigs-hdr@32473
  ...]` element and the hash-input rules from the raw-key mode apply
  identically in X.509 mode.
